我检索了当前最主流的多 Agent 产品前端设计（ClawPort、OpenClaw、LangSmith、MindStudio、FulmenAgent 等），总结出一个核心趋势：**现代 Agent 前端不再是"聊天窗口"，而是"任务驾驶舱"**——强调实时状态流、可视化编排、人机协同节点和成本/性能的可观测性。

结合你的 OmniOps 简化版流程（CSV/OCR → 感知 → 诊断 → 方案 → 校验 → 人工审核），我为你设计了一套**"案件卷宗式"**前端方案。

---

## 一、设计哲学：从"聊天框"到"案件卷宗"

| 传统设计 | 现代 Agent 设计（推荐） |
|---------|----------------------|
| 底部输入框 + 上方对话气泡 | **左侧 Session 列表** + **中央执行流水线** + **右侧详情抽屉** |
| 等 AI "想完"再一次性输出 | **流式节点状态**：每个 Agent 是一个"卡片节点"，实时显示思考中/已完成/需人工介入 |
| 纯文本输出 | **结构化渲染**：CSV 表格、拓扑图、方案卡片、审批按钮 |
| 用户被动等待 | **主动介入点**：在特定节点弹出人工审核面板，支持修改/确认/驳回 |

**参考来源**：ClawPort 的 Kanban 任务板与 Org Map ，MindStudio 的可视化 Workflow Builder 与执行追踪 ，以及 Microsoft 提出的 Multi-Agent 可见性原则（日志、可视化、性能指标）。

### 2.2 各区域设计详解

#### A. 左侧：Session 案件列表
- **状态标签**：`analyzing`（蓝色脉冲）/ `pending_human`（橙色警告）/ `resolved`（绿色）/ `escalated`（红色）
- **摘要卡片**：网元名 + 核心告警码 + 等待时间（如"NE-BJ-01 LINK_FAIL · 待审 4 分钟"）
- **筛选器**：按我的待审 / 处理中 / 今日已关闭

#### B. 中央：Agent 流水线画布（核心）
这是**最关键的创新点**。不用复杂的 DAG 拖拽编辑器（太重），而是用**垂直时间轴 + 节点卡片**：

```
┌────────────────────────────────────┐
│  📄 输入节点 (Perception)          │
│  CSV: alarm_20260429.csv           │
│  行数: 12  |  置信度: 0.98         │
│  [展开查看原始表格]                 │
└────────────┬───────────────────────┘
             ▼
┌────────────────────────────────────┐
│  🔍 诊断节点 (Diagnosis)  🔄 思考中...│
│  模型: DeepSeek-V3                 │
│  Token: 1,204 / 耗时: 4.2s         │
│  ─────────────────────────────     │
│  已生成结论: "K1SL64 光模块..."     │
│  [实时打字机效果显示推理过程]        │
└────────────┬───────────────────────┘
             ▼
┌────────────────────────────────────┐
│  📊 影响节点 (Impact)  ✅ 完成      │
│  影响链路: 2条  |  业务: 专线-A     │
│  [查看拓扑图]                       │
└────────────┬───────────────────────┘
             ▼
┌────────────────────────────────────┐
│  💡 方案节点 (Planning)  ✅ 完成    │
│  风险等级: Medium  |  需人工审核    │
└────────────┬───────────────────────┘
             ▼
┌────────────────────────────────────┐
│  ✓ 校验节点 (Verification) ✅ 完成  │
│  拓扑校验通过 | 约束检查通过        │
└────────────┬───────────────────────┘
             ▼
┌────────────────────────────────────┐
│  👤 人工审核 (Human Review) ⏸️ 阻塞 │
│  等待工程师确认...                   │
│  [采纳] [修改] [驳回] [转交]        │
└────────────────────────────────────┘
```

**设计要点**：
- **节点颜色**：灰色（等待）→ 蓝色脉冲（运行中）→ 绿色（完成）→ 橙色（需人工）→ 红色（异常）
- **流式思考**：诊断 Agent 运行中，节点内部实时显示 CoT 片段（参考 ChatGPT 打字效果）
- **点击穿透**：点击任意节点，右侧抽屉展开该 Agent 的完整输入/输出/工具调用

#### C. 右侧：详情抽屉（Context Inspector）
根据当前选中节点动态切换内容：
- **感知节点**：原始 CSV 表格（带格式）+ OCR 置信度标记（低置信度字段高亮黄色）
- **诊断节点**：RAG 召回的相似案例列表（Top-3）+ 知识库引用来源
- **方案节点**：结构化方案卡片（步骤列表、风险等级、所需工具）
- **人工审核节点**：审批表单 + 历史操作记录

## 四、技术架构选型

### 4.1 Demo 级（单容器 / 本地开发）

**目标**：`npm run dev` 一键启动，对接后端 Mock 或真实 FastAPI。

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **框架** | **Next.js 14+ (App Router)** | 全栈能力，API Routes 可直接对接后端，部署简单 |
| **语言** | TypeScript | 类型安全，前后端共享 Schema |
| **样式** | **Tailwind CSS + shadcn/ui** | 现成的深色主题、表格、卡片、Dialog、Sheet 组件 |
| **状态管理** | **Zustand** | 比 Redux 轻量，适合 Session 列表 + 当前选中 Session |
| **数据获取** | **SWR** | 自动轮询 + 缓存 + 重验证，适合实时性要求不高的 Demo |
| **表格渲染** | **TanStack Table** | CSV 结构化数据展示，支持排序/筛选 |
| **流式推送** | **SSE (EventSource)** | 基于 HTTP，穿透性好，Demo 阶段足够 |
| **图表/拓扑** | **React Flow** 或 **Mermaid** | Agent 流水线可视化，React Flow 可交互性更好 |

**Demo 级项目结构**：
```
frontend/
├── app/
│   ├── page.tsx              # 主布局（三栏）
│   ├── api/sse/route.ts      # SSE 连接后端（Next.js API Route）
│   ├── components/
│   │   ├── SessionList.tsx   # 左侧案件列表
│   │   ├── AgentPipeline.tsx # 中央流水线节点
│   │   ├── DetailDrawer.tsx  # 右侧详情抽屉
│   │   ├── AuditCard.tsx     # 人工审核卡片
│   │   └── UploadZone.tsx    # 上传入口
│   └── hooks/
│       ├── useSession.ts     # Zustand 状态
│       └── useSSE.ts         # SSE 连接管理
├── components/ui/            # shadcn/ui 组件
└── lib/
    ├── types.ts              # 共享类型（与后端对齐）
    └── api.ts                # API 客户端
```

### 4.2 生产级（分布式 / 企业部署）

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **框架** | **Next.js 15+ (App Router) + React 19** | Server Components 减少首屏 JS，Streaming SSR |
| **状态管理** | **Zustand + TanStack Query** | Zustand 管客户端 UI 状态，Query 管服务端状态 |
| **实时通信** | **WebSocket (Socket.io)** | 双工通信，支持工程师在审核时实时打字同步给后端 |
| **表格/大数据** | **TanStack Table + Virtuoso** | CSV 万行级虚拟滚动，不卡顿 |
| **拓扑可视化** | **React Flow + dagre** | 自动布局，支持拖拽查看 Agent 子图 |
| **图表** | **Recharts / Tremor** | 系统负载、Token 消耗、MTTR 趋势图 |
| **主题** | **next-themes** | 系统级深色模式，支持 Dark/Light/系统跟随 |
| **部署** | **Docker + Nginx** | Next.js standalone 输出，Nginx 做反向代理和静态资源缓存 |

**生产级关键优化**：

1. **前端状态与后端状态机严格同步**
   - WebSocket 推送 `session_state_change` 事件，Zustand 立即更新节点颜色
   - 乐观更新：工程师点击"采纳"后，前端立即将节点变为绿色，不等后端回执（提升感知速度）

2. **组件懒加载**
   - React Flow 画布、Recharts 图表用 `dynamic import`，首屏只加载核心列表和当前激活流水线

3. **Edge Function 鉴权**
   - 如果部署在 Vercel，用 Middleware 做 JWT 校验；自托管用 Nginx + Lua/OpenResty

4. **前端缓存策略**
   - Session 列表：SWR `refreshInterval: 5000`（5 秒轮询）
   - 知识库案例：TanStack Query `staleTime: Infinity`（极少变化）
   - 用户配置：localStorage + Zustand persist
